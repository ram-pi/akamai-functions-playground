02 — architecture agnosticism
KEY INSIGHT The .wasm binary has zero knowledge of CPU architecture.
It's the runtime's responsibility to translate bytecode to the host ISA.
🔷
my-module.wasm
same file · forever · no recompile needed
→
wasmtime on x86_64 emits x86 machine code
🖥️ Intel / AMD
→
wasmtime on ARM64 emits ARM machine code
📱 Apple M / AWS Graviton
→
wasmtime on RISC-V emits RISC-V machine code
🔬 embedded / SiFive
💡 The .wasm is bytecode for an abstract stack machine — no registers, no calling conventions, no ISA-specific instructions. The engine's JIT backend owns all of that. This is why a module compiled on your MacBook M3 runs unchanged on your x86_64 Linode nodes.
AOT exception — pre-compiled artifacts
Runtimes can optionally AOT-compile to a native artifact for faster cold starts — but that artifact is arch-specific and is purely an optimisation. The source .wasm remains the portable truth.
wasmtime compile --target x86_64 my-module.wasm -o my-module.cwasmarch-specific
03 — runtime execution model
🧱
Stack Machine
WASM uses a virtual stack machine. Instructions push/pop values. No registers exposed — all operations work on the evaluation stack, making validation simple and secure.
🧠
Linear Memory
Flat, contiguous byte array. Starts at 64KB pages. Grows via memory.grow. Completely isolated from JS heap — no pointer escape, no memory corruption outside the sandbox.
🛡️
Sandboxed Execution
No syscalls, no filesystem, no network access by default. Everything must be explicitly imported from the host. WASM can only do what the host allows — capability-based security.
🔀
JIT Compilation
Engines (V8, SpiderMonkey, Wasmtime) validate bytecode once, then JIT-compile to native machine code targeting the host CPU. Streaming compilation starts before full download.
04 — key properties
🌐
Portable
Same .wasm runs on browser, Node.js, Deno, Wasmtime, edge runtimes — any host with a WASM engine, any CPU arch.
⚡
Fast
Predictable performance. JIT to native, no GC pauses for WASM memory. 1.5–2× slower than native C at worst.
🔒
Secure
Formally verified type system. Memory bounds checked. No undefined behavior. Capability-based access model.
🗜️
Compact
Binary format is dense. Typical modules are 2–5× smaller than equivalent JS. Gzips well for fast delivery.
🔗
Interoperable
JS ↔ WASM calls are efficient. WIT (WASM Interface Types) enables richer cross-language data exchange.
🧩
Modular
Each module is self-contained with explicit imports/exports. WASI extends this to OS-level capabilities.
05 — spin & spinkube — wasm for the cloud
🌀
What is Spin?
Raw WASM gives you a fast, portable binary — but it has no opinion on how to serve HTTP requests, route traffic, or scale on Kubernetes. Spin (by Fermyon, now part of Akamai) is the framework that fills that gap. It wraps your .wasm module in a trigger-based runtime, turning it into a deployable microservice.
What Spin adds to the WASM flow
1
spin.toml — the manifest
Defines your app: which .wasm file handles which HTTP route, what environment variables it gets, what host capabilities it's allowed to use (key-value store, outbound HTTP, etc.).
2
Triggers — HTTP, Redis, cron
Spin listens for an event (an incoming HTTP request, a message on a queue) and instantiates your .wasm module just to handle that one request — then tears it down. True per-request isolation with near-zero cold start.
3
Built-in capabilities via WASI
Outbound HTTP, key-value stores, SQLite, Redis, variables — all exposed to your module as safe WASI interfaces. Your code never talks to the OS directly; Spin mediates everything.
4
SpinKube — Spin on Kubernetes
SpinKube brings a SpinApp CRD to your cluster. Instead of a container image, you ship a .wasm OCI artifact. The containerd-shim-spin on each LKE node runs it directly via Wasmtime — no Docker image, no OS layer inside the pod.
